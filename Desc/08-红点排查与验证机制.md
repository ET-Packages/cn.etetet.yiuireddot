# 红点排查与验证机制

## 1. 文档目标

这份文档只回答一件事：

- 当有人说“红点没生效”时，应该怎么判断

核心原则先写死：

- **不能因为用户说“我没看到效果”，就直接认定逻辑层没生效**
- 红点链路是多段组成的，任何一段断掉，最终都可能表现成“没看到红点”
- 所以判断红点问题，必须按固定检查机制逐层确认，不能靠猜

## 2. 红点完整链路

当前项目里，一条红点要真正被看到，至少要经过下面 5 步：

1. 定义红点 key
2. 配置 key 在红点树中的父子关系
3. UI 节点绑定这个红点 key
4. 业务逻辑正确设置该 key 的值
5. 运行时通过调试工具确认第 4 步到底有没有生效

这 5 步缺一不可。

## 3. 标准检查机制

以后排查红点问题，统一按这个顺序做。

### 第一步：先检查 key 是否存在

至少确认这 3 处是否一致：

- `Packages/cn.etetet.yiuireddot/Scripts/Model/Share/ERedDotKeyType.cs`
- `Assets/GameRes/RedDot/RedDotKeyAsset.asset`
- 相关业务代码里的 `ERedDotKeyType.xxx`

实际要确认的是：

- 业务代码使用的 key 是否真的存在
- 枚举值和描述是否对应
- 没有写错 id，没有拿错兄弟节点 key

如果这一步就错了，后面全都不用看。

### 第二步：检查 RedDotConfig 是否把 key 接进树

只定义 key 不够，还必须检查：

- `Assets/GameRes/RedDot/RedDotConfigAsset.asset`

这里要确认：

1. 当前 key 是否有配置
2. 父节点是否配置正确
3. 是否被接入目标红点链路
4. `SwitchTips` 等必要配置是否符合预期

如果这一步缺失，运行时即便设置了 count，也可能进不了完整红点树。

### 第三步：检查 UI 是否真的绑定了这个 key

这一步必须和“逻辑层是否生效”分开判断。

要确认：

1. 对应预制体或界面节点，是否挂了红点绑定组件
2. 绑定的是不是当前这个 key
3. 如果是动态绑定，运行时有没有真的执行 `ChangeBind(...)`
4. 绑定节点是不是当前肉眼观察的那个入口

如果没有这一步：

- 逻辑层可能已经设置成功
- 调试面板里也可能能看到 count
- 但 UI 上依然什么都没有

所以“肉眼没看到”不能直接推出“逻辑层没执行”。

UI 绑定规则参考：

- `06-红点UI绑定与预制体规范.md`
- `07-动态红点绑定.md`

### 第四步：检查业务逻辑是否真的设置了正确的 key 和 count

这一步才是典型的逻辑层问题。

要确认：

1. 业务代码设置的是不是目标 key
2. 设置时机对不对
3. 计算出来的 count 对不对
4. 数据变化后有没有重新刷新
5. 是不是在错误的生命周期里调用，导致运行时对象还没准备好

例如：

- key 写错
- count 计算错
- 数据先变了，但没有重新触发刷新
- 登录阶段对象未初始化，消息抛早了

这些才属于真正的逻辑层问题。

## 4. 第四步不能靠猜，必须有运行时判断点

这里只把话说死：

- **以后不能只因为“我没看到红点”就直接判断第 4 步有问题**

第 4 步必须通过运行时调试信息确认。

## 5. 运行时标准判断点

当前红点系统已经提供了编辑器 GM 调试能力。

重点参考：

- `RedDotPanel`
- `RedDotDataItem`
- `RedDotStackItem`
- `RedDotStackHelper`

这些能力的意义不是“再做一个 UI”，而是给出运行时真实判断点：

1. 当前 key 有没有被设置
2. 当前 key 的 count 是多少
3. 父子链路有没有正确汇总
4. 某个节点为什么亮、为什么不亮
5. 红点栈里的调试信息是否和预期一致

### 5.1 以后排查第 4 步的标准结论来源

以后要判断“逻辑层到底有没有写对”，标准结论来源应当是：

- 运行时调试面板
- `RedDotStackHelper` 提供的调试信息

而不是：

- 用户口头说“没亮”
- 肉眼看起来“像是没生效”
- 猜测某段逻辑“应该走了”

### 5.2 正确结论方式

正确的判断顺序是：

1. 先看 GM / 调试面板里该 key 有没有值
2. 再看父链路是不是也正确收到聚合结果
3. 最后再回头看 UI 节点有没有绑定和显示

这样才能区分：

- 是逻辑层没设置
- 是树配置没接通
- 是 UI 没绑定
- 还是绑定了错误节点

## 6. 调试面板支持手动设值

这里要单独强调一个很关键的能力：

- 红点调试能力不只是“查看当前值”
- **也可以手动给某个 key 设置值**

这个能力非常重要，因为它可以把“逻辑赋值问题”和“配置/绑定问题”硬切开。

### 6.1 人工排查方式

以后人工排查时，可以直接在红点调试面板里：

1. 选中目标 key
2. 人工设置一个明确的测试值，例如 `1`
3. 观察目标 UI 是否立刻出现对应红点表现

如果人工手动设值后：

- UI 仍然完全看不到变化

那就基本不用再优先怀疑第 4 步了。  
这种情况下，问题大概率就在前 3 步：

1. key 不对
2. config 没接进链路
3. UI 没绑定或绑错节点

也就是说：

- **人工手动设值都看不到 UI 变化，就先不要查业务逻辑赋值**

先把前 3 步查清楚再说。

### 6.2 反向判断逻辑

如果人工手动设值后：

- UI 立刻能看到正确变化

那就说明：

- key 本身存在
- config 链路基本通
- UI 绑定也基本正常

此时再去查第 4 步，也就是业务逻辑为什么没把这个值正确设进去，才是对的顺序。

## 7. AI 现在也支持主动设值调试

这一步不是人工专属。  
当前已经通过 `YIUIMCP` 扩展了红点相关原子操作，AI 可以直接读取红点运行时状态，也可以直接手动设置红点数量。

当前 AI 排查红点时，可以直接使用这两类能力：

1. 直接读取某个 red dot key 的运行时状态
2. 直接给某个 red dot key 手动设置一个测试值
3. 观察运行时调试信息和 UI 反馈
4. 再决定问题属于前 3 步还是第 4 步

这意味着以后 AI 排查红点，不必强依赖业务逻辑先跑通。

也就是说，AI 可以先绕开第 4 步，直接验证：

- 这个 key 能不能通过红点系统链路传到目标 UI

如果连人工/AI 手动设值都传不过去，那继续盯业务逻辑就是方向错了。

### 7.1 当前已接入的 MCP 原子操作

当前已经接入 2 个红点相关原子操作：

1. `GetRedDotDebugInfo`
2. `SetRedDotCount`

它们的定位如下：

- `GetRedDotDebugInfo`
  - 用途：读取某个 key 的当前运行时真相
  - 适合判断：
    - 当前 count 是多少
    - 当前 realCount 是多少
    - tips 是否开启
    - 是否有父子链路
    - 最近是谁改了它
    - 改成了多少

- `SetRedDotCount`
  - 用途：给某个 key 人工注入测试值
  - 适合判断：
    - UI / config / key 这前三步链路是否真的通
    - 某个业务红点是不是会被后续业务逻辑覆盖

### 7.2 当前 MCP 返回的是 JSON 字符串

这套红点原子操作当前返回的是 JSON 字符串，而不是面向人工排版的说明文本。

这样做的原因是：

- 更适合 AI 稳定解析
- 更适合后续自动化判断
- 不依赖人工去读大段描述文本

### 7.3 `GetRedDotDebugInfo` 当前可拿到的核心信息

基础模式至少会返回：

- `key`
- `desc`
- `exists`
- `count`
- `realCount`
- `tips`
- `switchTips`
- `parentKeys`
- `childKeys`
- `parentCount`
- `childCount`
- `stackCount`

完整模式还会追加：

- `stacks`
  - 每条包含：
    - `id`
    - `time`
    - `os`
    - `source`
    - `content`

其中最关键的是：

- `count`
  - 当前真正会显示出来的数量
- `realCount`
  - 不受 tips 开关影响的真实数量
- `tips`
  - 当前是否允许显示
- `stacks`
  - 谁改的、怎么改的、从哪条调用链改的

### 7.4 `SetRedDotCount` 当前的实际意义

`SetRedDotCount` 不只是“改个值看看”。

它在排查里的真正作用是：

1. 验证前 3 步链路是否通
2. 验证这个 key 会不会被后续业务逻辑重新覆盖

例如：

- AI 先把某个 key 设成 `1`
- 如果 UI 立即亮起，说明：
  - key 存在
  - config 大概率通
  - UI 绑定大概率也通
- 如果随后它又变回 `0`
  - 说明不是设值没成功
  - 而是被业务层后续刷新覆盖了

这个结论非常关键。

## 8. 标准排查顺序

以后统一按下面顺序排查，不要乱跳：

1. `ERedDotKeyType` 是否存在正确 key
2. `RedDotKeyAsset` 是否有对应描述
3. `RedDotConfigAsset` 是否正确接入父子链路
4. UI 预制体 / 动态节点是否绑定该 key
5. 先尝试人工或 AI 通过调试能力手动设置测试值
6. 观察 UI 是否有变化
7. 如果第 5~6 步通过，再检查业务逻辑是否正确设置实际 count
8. 如果第 5~6 步不通过，优先回头查前 3 步与 UI 绑定，不要先怀疑业务逻辑

这个顺序的核心目的是：

- 先排除“资源和配置层断链”
- 再排除“表现层没绑定”
- 再通过手动设值判断链路是否可达
- 最后才认定是逻辑层问题

## 9. 以后 AI / 开发默认执行规则

以后 AI 或开发者处理红点问题时，默认必须给出下面 3 类结论：

### 9.1 配置层结论

- key 是否存在
- config 是否接入

### 9.2 表现层结论

- UI 是否绑定
- 绑定的是不是正确 key

### 9.3 运行时逻辑结论

- GM / 调试面板里是否能看到该 key
- 当前 count 是多少
- 父链路聚合是否正确
- 人工或 AI 手动设值后，UI 是否能正确响应
- 如果手动设值后又被改回去，最后一次覆盖它的是谁

如果缺少第 9.3 条，就不能轻易下结论说“逻辑没生效”。

## 10. 当前 MCP 使用方式

当前这套红点 MCP 不是直接裸调 Unity MCP 端口，而是统一走 `UTO` 转发链路。

调用路径是：

1. 调用 `invoke-uto-tool.ps1`
2. 启动 `UTO HTTP Server`
3. 由 `UTO` 转发到 Unity MCP
4. Unity MCP 执行红点原子操作并返回结果

这意味着当前排查时，不要把“已经能通过 UTO 调到工具”与“Unity 业务逻辑正确”混为一谈。

`UTO` 打通只代表：

- 工具注册链路是通的
- AI 已经能拿到运行时红点真相

后面真正要判断的，仍然是：

- 当前 key 的状态到底是什么
- 是谁最后改了它
- 是手动测试值生效了，还是又被业务逻辑覆盖了

### 10.1 当前 MCP 的典型判定口径

当前使用 MCP 排查时，至少要区分下面几种情况：

#### 情况一：`SetRedDotCount` 后，UI 立刻亮起且值保持

说明：

- 前 3 步大概率没问题
- 当前也没有业务逻辑立刻把它覆盖掉

#### 情况二：`SetRedDotCount` 后，运行时一度变成目标值，但随后又被改回去

说明：

- 设值本身成功了
- 前 3 步大概率也是通的
- 真正的问题是后续业务逻辑把它重新刷掉了

这时要优先看 `GetRedDotDebugInfo` 的堆栈信息，确认最后一次覆盖是谁干的。

#### 情况三：`SetRedDotCount` 后，运行时值根本没有变化

说明：

- 要优先怀疑 key 不对、节点不可直接设值、或管理器状态不对
- 不要先下结论说 UI 没绑定

#### 情况四：运行时值变了，但 UI 没变化

说明：

- 优先查前 3 步
- 尤其查：
  - config 是否接树
  - UI 是否真的绑定这个 key
  - 当前看的 UI 节点是不是目标节点

## 11. 一句话总结

红点问题不能只看最后有没有亮。  
必须把它拆成：

- key 定义
- config 接树
- UI 绑定
- 业务赋值
- 运行时调试确认

其中第 4 步不能靠猜，必须通过 `RedDotPanel`、`RedDotDataItem`、`RedDotStackItem`、`RedDotStackHelper`，以及当前已接入的 `GetRedDotDebugInfo / SetRedDotCount` MCP 原子操作做实际判断。
并且在必要时，应优先通过人工或 AI 手动设值，先验证前 3 步和 UI 链路是否通，再决定要不要继续查第 4 步。
