# AI 新增红点工作流

## 1. 文档目的

以后如果用户只说一句：

- “这里加个红点”
- “任务这里要亮一下”
- “伙伴新解锁要提示”

AI 不能直接开改。  
必须先按这份工作流把缺失信息补齐，再决定改动范围。

## 2. AI 默认工作顺序

### 第一步：先判断这是哪一类红点

至少先分清：

- 奖励可领取型
- 重要信息待查看型
- 阈值状态提醒型

如果连这一步都没分清，就不要开始配 key。

并且还要先判断：

- 这是稳定聚合层级
- 还是动态末级对象

如果是动态末级对象，默认**不要新增枚举 key**，而是由上一级稳定节点承接红点数量或状态。

### 第二步：先判断是“新增节点”还是“复用已有节点”

AI 必须先查现有树：

- `ERedDotKeyType.cs`
- `RedDotKeyAsset.asset`
- `RedDotConfigAsset.asset`

不要因为用户没说，就默认要新建 key。

### 第三步：先判断这次是不是只改配置

AI 必须明确判断下面哪一种：

1. 只需补 key 和父子关系
2. key + 配置 + 业务触发
3. key + 配置 + 已查看状态
4. key + 配置 + UI 监听

如果需求里出现下面这些词，通常就不是“只改配置”：

- 已读
- 已查看
- 打开过
- 切换过
- 领取后
- 阈值恢复后

### 第四步：新增 key 时必须按 ID 规则分配

规则见：

- `02-红点Key与ID规则.md`

AI 不能再用短递增 ID。

同时要记住：

- key 只加到稳定层级
- 不要给“未知数量、动态生成、运行时变化的具体对象”硬加固定 key

### 第五步：至少同步这 3 处

新增 key 时，最少同时修改：

1. `Scripts/Model/Share/ERedDotKeyType.cs`
2. `Assets/GameRes/RedDot/RedDotKeyAsset.asset`
3. `Assets/GameRes/RedDot/RedDotConfigAsset.asset`

只改其中 1~2 处，通常就是半残状态。

### 第六步：如果改了 `.cs`，必须编译验证

命令：

```powershell
powershell -ExecutionPolicy Bypass -Command "& 'Packages/cn.etetet.yiuimcp/Config/compile-unity-flow.ps1' -Force 0"
```

不能拿 `dotnet build` 替代。

## 3. AI 新增红点时必须先回答的问题

如果用户没说清，AI 至少要自己先补出这几个判断：

1. 红点挂在哪个父节点下
2. 它是聚合节点还是叶子节点
3. 客户端真实状态来源是什么
4. 红点亮起条件是什么
5. 红点消除条件是什么
6. 是否需要“已查看态”
7. 是否已有现成业务组件承接这个状态
8. 这次红点数量是谁算，算完由谁调用 `SetCount`

## 4. AI 不允许做的事

### 4.1 不允许把 MessageObject 当红点临时状态

这是禁区，原因你项目总 AGENTS 已经写死了。

### 4.2 不允许把业务状态塞进 YIUI Panel/View

UI 只负责表现，不负责业务缓存。

### 4.3 不允许看见“打开界面时请求”就让 UI 自己发协议

需要落到客户端业务组件里，由组件控制是否请求、是否节流、是否缓存。

### 4.4 不允许为了快，直接跳过 `ERedDotKeyType`

运行时 key 来源就是它，绕不过去。

### 4.5 不允许在 UI 打开时临时计算业务红点

错误方向：

- 打开任务面板时顺手遍历任务列表，再去决定 `任务_主线` 红点有几个
- 刷任务 item 预制体时顺手回推某个父级红点数量

正确方向：

- 任务系统数据层在任务变化时计算主线红点数量
- 计算完成后统一调用红点赋值
- UI 打开时只读取红点结果并显示

## 5. AI 回复用户时的最小结构

当用户说“加红点”时，AI 至少应该给出下面这种判断结果：

1. 这次红点归属的树位置
2. 这次是否新增 key
3. 新 key 的建议 id 与描述
4. 这次只改配置还是还要补业务状态
5. 后续验证方式

## 6. 一个合格的例子

用户说：

- “邮件有未读或附件没领时，菜单那里要亮”

AI 不应直接答：

- “我去加个红点”

AI 应该先得出：

1. `菜单` 已有二级节点
2. `菜单_邮件` 已有三级节点
3. 如果只是菜单入口聚合，可能复用 `菜单_邮件`
4. 若要更细粒度，还需要叶子，如：
   - `菜单_邮件_未读邮件`
   - `菜单_邮件_附件可领取`
5. 这不是纯配置问题，因为邮件状态要来自客户端邮件业务组件

这才是有效工作流。
