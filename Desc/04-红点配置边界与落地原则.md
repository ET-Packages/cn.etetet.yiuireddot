# 红点配置边界与落地原则

## 1. 什么叫“只改配置就够”

只有当下面三件事都已经存在时，才可能只改配置：

1. 运行时 key 已存在
2. 客户端状态来源已存在
3. 业务触发链路已存在

这时候你可能只是在补：

- 父子关系
- 聚合层级
- 显示描述

## 2. 什么情况一定不止改配置

### 2.1 需要“已查看态”

典型词：

- 打开过
- 看过
- 切换过
- 点击过

这类需求本质上都要有状态承载。

### 2.2 需要“已领取态”

如果客户端本地没有稳定镜像，就必须补业务同步或业务组件。

### 2.3 需要“未读态”

如果未读是服务端语义，客户端就要有明确镜像来源。  
不要拿某个界面是否打开过，去代替服务端未读。

### 2.4 需要阈值判断

例如：

- 总资产低于阈值时亮

这类需求通常依赖现成 Numeric 或业务组件，不是红点包自己判断出来的。

## 3. 叶子与聚合的原则

### 3.1 叶子节点

这里要分清“红点树叶子节点”和“业务里的最终具体对象”不是一回事。

红点树里的叶子节点，指的是**稳定的、可声明 key 的最末层聚合节点**。  
业务里的最终具体对象，很多时候是动态的，不应该直接映射成固定 red dot key。

例如：

- `任务_主线`
- `菜单_邮件`
- `伙伴_新伙伴`

这些是红点系统里的稳定节点。

但下面这些通常不是固定 key：

- 某一条具体主线任务
- 某一封具体邮件
- 某一个具体公告页签

这些属于业务动态对象，应由业务层计算后汇总到稳定红点节点。

### 3.2 聚合节点

聚合节点不发明状态，只聚合子节点。

例如：

- `任务`
- `菜单`
- `伙伴`

## 4. 红点系统与业务系统的职责边界

### 4.1 红点系统负责什么

红点系统负责：

- 承接固定 key
- 维护父子聚合关系
- 存储当前红点数量或开关结果
- 对外提供统一显示结果

### 4.2 红点系统不负责什么

红点系统不负责：

- 遍历业务数据得出“应该亮几个”
- 决定某条任务是否可领取
- 决定某封邮件是否未读
- 决定某个伙伴是否属于“新解锁未查看”

这些都应该由业务数据层或业务组件自己算完，再把结果喂给红点系统。

### 4.3 UI 层负责什么

UI 层负责：

- 读取某个 key 当前是否显示
- 把红点挂到具体按钮、列表项、入口图标上
- 在创建具体预制体时，把“这条动态业务对象是否显示红点”表现出来

### 4.4 UI 层不负责什么

UI 层不负责：

- 重新计算业务红点数量
- 在打开界面时临时决定父级红点数
- 维护业务红点缓存
- 倒推修改红点数据层

## 5. 任务系统示例

以任务系统为例：

- 可声明 key：
  - `主界面`
  - `任务`
  - `任务_主线`

- 不声明 key：
  - 某一条具体任务

正确做法：

1. 任务数据层在任务变化时，自己计算当前可领取的主线任务数量
2. 计算完成后，调用红点赋值给 `任务_主线`
3. `任务` 和 `主界面` 由红点树自动聚合
4. 具体 UI 打开后，某条任务 item 是否显示红点，由表现层根据那条任务自己的业务状态来显示

错误做法：

1. 打开任务 UI
2. 遍历当前界面上有哪些任务 item
3. 临时算出 `任务_主线` 红点数量
4. 顺手把父级红点也一起改掉

这种写法把业务、红点、UI 三层搅在一起，后面一定失控。

## 6. `SwitchTips` 怎么理解

- `true`：玩家可以手动开关提示
- `false`：不可关闭，永久提示

不要乱用 `false`。  
只有业务上明确要求“这个红点不允许用户关闭”时才这么配。

## 7. 推荐落地顺序

1. 先定树层级
2. 再定 key 和描述
3. 再定父子依赖
4. 再查客户端状态来源
5. 再补业务触发与消除
6. 最后验证 UI 是否按预期显示

## 8. 判断一句话

如果一个红点需求描述里带“查看”“读取”“打开过”“点击过”“切换过”，  
默认先怀疑它不是纯配置问题。
