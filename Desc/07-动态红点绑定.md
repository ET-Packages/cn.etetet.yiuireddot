# 动态红点绑定

## 1. 文档目标

这份文档只讲一件事：

- 不提前修改业务 UI prefab
- 运行时直接给某个 UI 节点挂一个红点

它不负责解释：

- 红点 key 怎么设计
- 红点数量由谁计算
- 红点树父子关系怎么配置

这些内容看前面的专题文档。

## 2. 动态红点解决什么问题

传统 UI 红点做法是：

- 在业务 prefab 上提前挂 `RedDotBind` / `RedDotTextBind` / `RedDot_TMP`
- 在 Inspector 里手动选择绑定 key

这种方式适合稳定 UI，但不适合下面这些场景：

- AI 只改代码，不方便进 prefab 手动挂脚本
- 某个节点是动态创建出来的
- 循环列表里某一项临时要显示红点
- 只是想给某个 UI 节点快速挂一个红点，不想改原 prefab

动态红点就是为这些场景补的运行时入口。

## 3. 当前实现思路

当前动态红点不是走 YIUI 动态 `Component/CDE` 实例化链。

真实流程是：

1. 调用侧传入目标 `Transform`、`ERadDotShowType`、红点 key 或手动数量。
2. `RedDotMgr` 根据 `ERadDotShowType` 找到对应的动态红点 prefab 资源名。
3. `RedDotMgr` 调用 `YIUIGameObjectPool.Inst.Get(resName, parent)` 从对象池取红点对象。
4. 红点对象挂到传入的 `parent` 下。
5. `RedDotMgr` 设置 9 宫格锚点。
6. `RedDotMgr` 从对象上获取 `IDynamicRedDotControl`。
7. key 模式调用 `BindKey(key)`，手动模式调用 `SetManualCount(count)`。
8. 不需要显示时，调用 `RemoveDynamicRedDot(gameObject)` 主动回收。

职责边界：

- `RedDotMgr` 负责动态加载、设置锚点、设置红点控制状态。
- `YIUIGameObjectPool` 负责对象池复用和自动回收。
- 红点 prefab 自己的脚本负责具体显示效果。
- 业务层负责计算红点数量，不要把业务判断塞到 UI 红点表现层里。

## 4. 动态红点显示类型配置

动态红点 prefab 不在调用代码里直接写资源名，而是通过 `ERadDotShowType` 关联。

相关文件：

- 枚举：`Packages/cn.etetet.yiuireddot/Runtime/Data/ERadDotShowType.cs`
- 配置资源：`Packages/cn.etetet.yiuireddot/Assets/GameRes/RedDot/RedDotShowAsset.asset`
- 配置数据：`Packages/cn.etetet.yiuireddot/Runtime/Show/RedDotShowData.cs`
- 编辑器入口：YIUI 红点工具里的 `红点动态预制关联` 页签

配置数据包含：

- `EnumName`：枚举名，也是运行时查找 key，必须唯一。
- `Des`：描述，会生成到 `ERadDotShowType` 的 `LabelText`。
- `ResName`：预制体资源名，运行时用它从 `YIUIGameObjectPool` 加载。

配置规则：

- 不需要写 ID。
- `EnumName` 必须全英文、大驼峰、唯一。
- 关联 prefab 的资源名必须全局唯一。
- 编辑器里可以拖 prefab，但实际存储的是 prefab 名称。
- 重新打开工具时，会根据 `ResName` 反查 prefab；找不到说明资源丢失。

调用侧只允许知道 `ERadDotShowType`，不要直接写 prefab 名。

## 5. 关键类型

### 5.1 `ERadDotShowType`

表示“我要使用哪一种红点显示样式”。

示例：

```csharp
ERadDotShowType.RedDot
ERadDotShowType.RedDotTMP
ERadDotShowType.RedDotText
ERadDotShowType.RedDotAnim
```

这个枚举由 YIUI 红点工具自动生成，请不要手动修改。

### 5.2 `ERedDotAnchor`

控制动态红点挂到父节点后的锚点位置。

当前支持 9 宫格：

- `Center`
- `LeftTop`
- `Top`
- `RightTop`
- `Left`
- `Right`
- `LeftBottom`
- `Bottom`
- `RightBottom`

默认值：

- `RightTop`

当前没有 offset。位置不合适时，优先调整父节点或红点 prefab 自身结构，不要随便扩展无限偏移参数。

### 5.3 `IDynamicRedDotControl`

动态红点 prefab 必须能提供统一控制入口。

当前接口：

```csharp
public interface IDynamicRedDotControl
{
    void BindKey(int key);
    void SetManualCount(int count);
    void ResetState();
    GameObject GetOwnerGameObject();
}
```

要求：

- 动态加载出来的对象必须能获取到 `IDynamicRedDotControl`。
- 获取不到会报错，并把对象放回对象池。
- 当前基础实现是 `RedDotBind`，它已经实现了这个接口。

## 6. API 使用方法

API 文件：

- `Packages/cn.etetet.yiuireddot/Runtime/Mgr/RedDotMgr_Dynamic.cs`

### 6.1 根据稳定 key 动态绑定

```csharp
public async ETTask<GameObject> BindDynamicRedDotByKey(
    Transform parent,
    ERadDotShowType showType,
    int key,
    ERedDotAnchor anchor = ERedDotAnchor.RightTop)
```

适用场景：

- 目标节点对应的是稳定红点 key。
- 红点数量由红点系统根据 key 自动监听刷新。
- 只是希望运行时挂上去，不想改 UI prefab。

已经验证可用的调用方式：

```csharp
RedDotMgr.Inst.BindDynamicRedDotByKey(
    self.u_ComPassRectTransform,
    ERadDotShowType.RedDotAnim,
    ERedDotKeyType.Key1002).Coroutine();
```

如果后续需要主动回收，应该保留返回的 `GameObject`：

```csharp
var redDotGo = await RedDotMgr.Inst.BindDynamicRedDotByKey(
    self.u_ComPassRectTransform,
    ERadDotShowType.RedDotAnim,
    ERedDotKeyType.Key1002);
```

指定锚点：

```csharp
var redDotGo = await RedDotMgr.Inst.BindDynamicRedDotByKey(
    self.u_ComPassRectTransform,
    ERadDotShowType.RedDotAnim,
    ERedDotKeyType.Key1002,
    ERedDotAnchor.RightTop);
```

注意：

- `ERedDotKeyType` 当前是静态常量类，不是 C# enum。
- 所以 API 参数是 `int key`，传 `ERedDotKeyType.Key1002` 是合法的。
- `key <= 0` 会直接报错并返回 `null`。

### 6.2 手动显示动态红点

```csharp
public async ETTask<GameObject> BindDynamicRedDotManual(
    Transform parent,
    ERadDotShowType showType,
    int count,
    ERedDotAnchor anchor = ERedDotAnchor.RightTop)
```

适用场景：

- 具体某条任务
- 具体某封邮件
- 具体某个列表项
- 具体某个格子
- 这类动态末级对象不适合进入稳定红点 key 枚举树

示例：

```csharp
var redDotGo = await RedDotMgr.Inst.BindDynamicRedDotManual(
    itemRectTransform,
    ERadDotShowType.RedDot,
    1);
```

文本红点示例：

```csharp
var redDotGo = await RedDotMgr.Inst.BindDynamicRedDotManual(
    itemRectTransform,
    ERadDotShowType.RedDotTMP,
    rewardCount,
    ERedDotAnchor.RightTop);
```

注意：

- `count >= 1` 才有显示意义。
- `count <= 0` 会报错并返回 `null`。
- 如果是无文本红点，通常只关心 `count > 0` 显示、`count <= 0` 不创建或回收。
- 手动模式的数量由业务层算好，红点系统不负责推导这个 count。

### 6.3 主动回收

```csharp
public bool RemoveDynamicRedDot(GameObject gameObject)
```

示例：

```csharp
RedDotMgr.Inst.RemoveDynamicRedDot(redDotGo);
```

说明：

- 内部会先调用 `IDynamicRedDotControl.ResetState()`。
- 然后调用 `YIUIGameObjectPool.Inst.Put(gameObject)` 回收到对象池。
- 传 `null` 会返回 `false`。
- 传入的对象不是红点对象时，会打错误日志。

## 7. 锚点规则

动态红点加载完成后，会统一设置：

- `anchorMin`
- `anchorMax`
- `pivot`
- `anchoredPosition = Vector2.zero`

当前实现里：

- `anchorMin` 和 `anchorMax` 根据 `ERedDotAnchor` 设置。
- `pivot` 固定为 `0.5, 0.5`。
- 本地坐标固定为 `0,0`。

如果红点位置看起来不合适，优先处理：

- 父节点的 RectTransform。
- 红点 prefab 自身尺寸和中心点。
- 选择更合适的 `ERedDotAnchor`。

## 8. 对象池边界

动态红点明确依赖：

- `YIUIGameObjectPool`

当前设计已经把对象生命周期交给对象池：

- 红点系统不维护 `handleId`。
- 红点系统不维护动态实例表。
- 红点系统不额外挂生命周期守卫脚本。

这意味着：

- 常规 UI 生命周期下，对象池可以自动处理一部分回收。
- 如果业务明确知道不需要这个红点了，建议主动调用 `RemoveDynamicRedDot`。
- 主动回收能让状态更明确，也能避免对象在界面频繁开关时残留到下一次复用。

## 9. 动态绑定和静态绑定怎么选

### 9.1 静态 UI 绑定

做法：

- 在 prefab 上提前放好红点节点。
- 挂 `RedDotBind` / `RedDotTextBind` / `RedDot_TMP`。
- Inspector 里直接选择红点 key。

优点：

- UI 配置更直观。
- 非程序也可以调整位置、样式、绑定对象。
- 修改绑定不一定需要程序介入。
- 对长期稳定 UI 更容易维护。

缺点：

- 需要修改 prefab。
- AI 不方便直接介入。
- 动态列表项和临时节点不方便提前配置。

### 9.2 动态红点绑定

做法：

- 在代码里调用 `RedDotMgr.Inst.BindDynamicRedDotByKey(...)`。
- 或调用 `RedDotMgr.Inst.BindDynamicRedDotManual(...)`。

优点：

- AI 可以介入，直接加一行代码就能挂红点。
- 不需要提前修改业务 UI prefab。
- 适合动态节点、循环列表项、临时挂点。
- 同一个 UI 节点可以按运行时条件决定是否挂红点。

缺点：

- 没有 UI 静态绑定灵活。
- 位置和样式调整主要依赖程序参数、父节点和红点 prefab。
- 如果只是改绑定 key 或微调 UI，静态绑定可以在编辑器里处理，不需要程序接入。
- 调用侧要自己管理返回的 `GameObject`，需要主动回收的场景必须留引用。

一句话：

- 稳定 UI 优先静态绑定。
- 动态 UI、AI 自动落代码、循环列表项优先动态绑定。

## 10. 常见使用场景

### 10.1 通行证按钮挂稳定 key 红点

```csharp
RedDotMgr.Inst.BindDynamicRedDotByKey(
    self.u_ComPassRectTransform,
    ERadDotShowType.RedDotAnim,
    ERedDotKeyType.Key1002).Coroutine();
```

这种写法适合：

- 打开 UI 时挂上红点。
- 不关心手动回收。
- 依赖 UI / 对象池生命周期处理回收。

### 10.2 需要主动回收的稳定 key 红点

```csharp
private GameObject m_PassRedDot;

m_PassRedDot = await RedDotMgr.Inst.BindDynamicRedDotByKey(
    self.u_ComPassRectTransform,
    ERadDotShowType.RedDotAnim,
    ERedDotKeyType.Key1002);

RedDotMgr.Inst.RemoveDynamicRedDot(m_PassRedDot);
m_PassRedDot = null;
```

### 10.3 列表项手动红点

```csharp
if (canReceive)
{
    await RedDotMgr.Inst.BindDynamicRedDotManual(
        itemRectTransform,
        ERadDotShowType.RedDot,
        1);
}
```

这里的 `canReceive` 应该来自业务数据层，而不是 UI 自己临时猜。

## 11. 必须记住的限制

### 11.1 动态红点不声明末级业务对象 key

例如任务系统：

- 稳定红点 key 可以到 `任务_主线`。
- 具体某一条任务不应该声明成稳定 key。

具体任务是否可领取，属于动态业务对象表现：

- 列表项自己根据业务数据决定是否调用 `BindDynamicRedDotManual`。

### 11.2 动态红点不负责业务计算

红点系统只负责显示。

下面这些事情不归动态红点系统管：

- 某条任务是否可领取。
- 某封邮件是否未读。
- 某个奖励是否可领取。
- 某个格子是否需要提示。

这些都应该由业务数据层算好，再把结果传给红点显示层。

### 11.3 `showType` 没有配置资源一定失败

如果 `ERadDotShowType` 没有在 `红点动态预制关联` 中配置对应 prefab，动态绑定会失败。

常见错误：

- 枚举生成了，但 `RedDotShowAsset` 没有对应数据。
- 配置里存的 `ResName` 对应 prefab 被改名或删除。
- prefab 名不是全局唯一。
- prefab 上没有实现 `IDynamicRedDotControl` 的脚本。

## 12. 给 AI 的调用口径

以后如果要让 AI 给某个 UI 节点加动态红点，需求里尽量给这些信息：

- 目标 UI 节点，也就是传哪个 `Transform`。
- 使用哪种 `ERadDotShowType`。
- 使用稳定 key，还是手动 count。
- 如果是稳定 key，给出 `ERedDotKeyType.xxx`。
- 如果是手动模式，说明 count 从哪个业务数据来。
- 是否需要指定 `ERedDotAnchor`。
- 是否需要主动回收并保存返回的 `GameObject`。

示例口径：

```text
给通行证按钮 self.u_ComPassRectTransform 挂动态红点。
样式用 ERadDotShowType.RedDotAnim。
key 用 ERedDotKeyType.Key1002。
默认右上角，不需要主动回收。
```

AI 应该落成：

```csharp
RedDotMgr.Inst.BindDynamicRedDotByKey(
    self.u_ComPassRectTransform,
    ERadDotShowType.RedDotAnim,
    ERedDotKeyType.Key1002).Coroutine();
```

## 13. 一句话总结

动态红点的本质是：

- **从 `YIUIGameObjectPool` 借一个红点对象**
- **挂到指定 UI `Transform` 下**
- **通过 `IDynamicRedDotControl` 设置 key 或 count**
- **不需要时再主动放回对象池**

它不是新的红点业务系统，只是给现有红点表现层补了一个运行时动态挂载入口。
