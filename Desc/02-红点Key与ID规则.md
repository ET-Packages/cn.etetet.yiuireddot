# 红点 Key 与 ID 规则

## 1. 先记住结论

红点 key 的 id 不走"简单递增"，而走"分段层级编码"。

当前规则：

- 根节点：`1`
- 第二级：在根后预留 `3` 位
- 第三级开始：每往下一层追加 `2` 位

这样做的目的不是省事，而是为了：

- 一眼看出层级归属
- 给每层留足扩展空间
- 让编辑器下拉可以按层级自动分段显示

## 2. 编码规则

### 2.1 根节点

固定：

- `1 = 根节点`

### 2.2 第二级

格式：

- `1 + 3位`

示例：

- `1001 = 系统A`
- `1002 = 系统B`
- `1003 = 系统C`

含义：

- `001`、`002`、`003` 是 `根节点` 下的二级系统序号
- 这一级直接预留 `999` 个系统

### 2.3 第三级

格式：

- `二级key + 2位`

示例：

- `100101 = 系统A_功能1`
- `100102 = 系统A_功能2`
- `100201 = 系统B_功能1`
- `100301 = 系统C_功能1`

含义：

- 每个二级节点下面默认再预留 `99` 个直接子节点

### 2.4 第四级及以后

格式：

- `父key + 2位`

示例：

- `10010201 = 系统A_功能2_子项1`
- `10010202 = 系统A_功能2_子项2`
- `10010301 = 系统A_功能3_子项1`

含义：

- 每一层都继续按 `99` 个直接子节点扩展
- 子项挂在对应父级功能节点下

## 3. 描述命名规则

### 3.1 二级描述

二级节点可以直接写系统名称：

- `系统A`
- `系统B`
- `系统C`

### 3.2 三级及以后描述

从三级开始，描述必须带完整父级前缀。

正确示例：

- `系统A_功能1`
- `系统A_功能2_子项1`
- `系统A_功能3_子项1`
- `系统B_功能1`
- `系统C_功能1`

错误示例：

- `功能1`
- `子项1`
- `功能3`

原因很简单：

- 编辑器看下拉时，要靠描述快速判断层级
- 调试红点树时，也要靠描述肉眼判断归属

## 4. 新增 key 的决策顺序

每次新增前按这个顺序判断：

1. 这次需求是新增系统入口，还是已有系统下的新子节点
2. 它的直接父节点是谁
3. 它应该挂在第几层
4. 同级里给它分配哪个序号
5. 描述是不是需要带完整父级前缀

## 5. 示例

### 5.1 在"系统A_功能3"下新增"子项1"

父级：

- `100103 = 系统A_功能3`

新 key：

- `10010301 = 系统A_功能3_子项1`

### 5.2 在"系统A_功能2"下新增"子项2"

父级：

- `100102 = 系统A_功能2`

新 key：

- `10010202 = 系统A_功能2_子项2`

### 5.3 在"系统B"下新增"功能1"

父级：

- `1002 = 系统B`

新 key：

- `100201 = 系统B_功能1`

## 6. 强制要求

- 不要为了省事回退到 `11/12/13` 这种短 ID。
- 不要跨层偷号。
- 不要把两个不同父级下的同义节点混用一个 key。
- 不要写没有父级前缀的三级及以后描述。
