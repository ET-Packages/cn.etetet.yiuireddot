# 红点基础规则与职责边界

## 1. 先把一句话说死

红点系统不是业务判断系统。  
红点系统是**业务结果的承载与聚合系统**。

## 2. 红点枚举可以到哪一步

红点 key 只能声明到**稳定层级**，不能声明到**动态未知层级**。

### 2.1 可声明的层级

例如任务系统：

- `主界面`
- `任务`
- `任务_主线`

这些都是稳定的、长期存在的层级，可以声明固定 key。

### 2.2 不可声明的层级

例如：

- 某一条具体任务
- 某一封具体邮件
- 某一个具体公告
- 某一个具体伙伴实例项

这些对象都可能是：

- 动态生成
- 数量不固定
- 运行时变化
- 未来配置扩展后继续增长

所以不能为它们逐个声明固定枚举 key。

## 3. 为什么不能给动态末级声明 key

原因不是“麻烦”，而是模型本身就错。

### 3.1 数量未知

具体业务对象可能无限扩展，枚举不应该追着动态数据跑。

### 3.2 生命周期不稳定

动态对象可能今天有、明天删、后天重建。  
把这种对象绑成固定 key，只会把枚举体系拖乱。

### 3.3 会污染红点树职责

红点树应该描述“稳定入口和稳定聚合关系”，而不是替业务对象建镜像数据库。

## 4. 数据层与表现层的边界

### 4.1 数据层负责什么

数据层负责：

- 监听业务状态变化
- 计算当前红点数量或是否显示
- 把计算结果赋值给固定 key

例如任务系统：

- 主线任务变化时
- 任务数据层重新计算“可领取主线任务数量”
- 然后调用红点赋值给 `任务_主线`

### 4.2 表现层负责什么

表现层负责：

- 打开 UI 时读取当前红点结果
- 在具体按钮、页签、预制体上显示
- 对某个具体动态对象做视觉表现

例如：

- 某条任务 item 是否显示小红点
- 某封邮件列表项是否显示未读点

这属于表现层，不属于红点数据层。

### 4.3 表现层不该做什么

表现层不该：

- 打开界面后顺手重新计算业务红点数量
- 根据当前界面展示内容倒推出父级红点数量
- 自己维护一套红点业务缓存

## 5. 红点系统到底管什么

红点系统管：

- key
- 父子关系
- 聚合
- 当前数量
- 是否显示
- 提示开关

红点系统不管：

- 业务规则本身
- 具体动态对象生命周期
- 某个系统“为什么应该亮”
- 哪条任务、哪封邮件、哪个伙伴算新内容

## 6. 任务系统示例

### 6.1 正确模型

固定 key：

- `主界面`
- `任务`
- `任务_主线`

动态对象：

- 具体任务 A
- 具体任务 B
- 具体任务 C

流程：

1. 任务系统数据层感知任务状态变化
2. 重新计算当前“可领奖主线任务数量”
3. 结果写入 `任务_主线`
4. 红点系统自动聚合到 `任务` 和 `主界面`
5. UI 打开时，只读取并显示

### 6.2 错误模型

错误写法：

1. 打开任务 UI
2. 看当前刷出来几个任务 item
3. 用 UI 当前看到的数据临时算红点
4. 再去写 `任务_主线`

这个写法的问题：

- UI 没开时状态不完整
- 列表没刷全时结果不可信
- 数据层和表现层彻底耦死

## 7. 对 AI 的强约束

以后 AI 收到红点需求时，必须先判断：

1. 这次新增的是稳定节点，还是动态对象
2. 如果是动态对象，是否应该停在上一层固定 key
3. 红点数量由哪个业务组件计算
4. UI 只是展示，还是用户在混淆数据层和表现层

只要用户描述落到“某条具体任务”“某封具体邮件”这种动态末级，AI 默认先反驳“给它单独加枚举 key”这条路。
