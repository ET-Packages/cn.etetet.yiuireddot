# 红点系统总览

## 1. 目标

红点系统的职责很单纯：

- 用统一树结构表示“哪里有奖励可领”或“哪里有重要信息待看”
- 用叶子节点承接真实业务状态
- 用父级节点做聚合显示

不要把红点系统理解成“业务状态本体”。  
红点只消费业务状态，不负责发明业务状态。

## 2. 当前资源分工

### 2.1 `ERedDotKeyType`

路径：

- `Packages/cn.etetet.yiuireddot/Scripts/Model/Share/ERedDotKeyType.cs`

职责：

- 定义所有红点 key 常量
- 运行时通过反射读取这里的 key 列表

结论：

- 新增 key 时，**不改这里就等于运行时没有这个红点**

### 2.2 `RedDotKeyAsset`

路径：

- `Packages/cn.etetet.yiuireddot/Assets/GameRes/RedDot/RedDotKeyAsset.asset`

职责：

- 存 `key -> 描述`
- 给编辑器和调试界面看

结论：

- 它不是运行时 key 的唯一来源
- 但如果不补它，编辑器和调试显示会不完整

### 2.3 `RedDotConfigAsset`

路径：

- `Packages/cn.etetet.yiuireddot/Assets/GameRes/RedDot/RedDotConfigAsset.asset`

职责：

- 存 `key -> ParentList`
- 配置红点树依赖关系
- 配置 `SwitchTips`

结论：

- 它决定“谁挂谁下面”
- 不负责“什么时候亮”

## 3. 红点需求拆分原则

每条红点需求都先拆成三件事：

1. 红点树位置
2. 触发来源
3. 消除条件

### 3.1 红点树位置

先回答：

- 这是根节点、聚合节点还是叶子节点
- 它属于哪个父级
- 它和同级节点是什么关系

### 3.2 触发来源

再回答：

- 红点由哪个客户端业务组件提供状态
- 是数量型、布尔型还是“已查看/未查看”型
- 有没有现成消息链路

### 3.3 消除条件

最后回答：

- 是领取后消失
- 已读后消失
- 打开后消失
- 切换查看后消失
- 还是阈值恢复后自动消失

## 4. 不要犯的错

### 4.1 只配树，不补状态来源

这种改法表面看像“红点加上了”，实际上只是把空节点挂树上。

### 4.2 给动态末级对象硬声明红点 key

这是典型错误。

例如任务系统里：

- `主界面`
- `任务`
- `任务_主线`

这些可以声明成固定 key。  
但“某一条具体主线任务”通常是动态数据，数量不固定、内容未知、运行时变化，所以**不应该为每个具体任务声明一个固定 key**。

红点枚举应该停在“稳定的聚合层级”。

### 4.3 把 UI 状态当业务状态

例如：

- 打开某个界面时顺手存在 Panel 上
- 点击某个按钮后只改 UI 本地字段

这类做法后面一定会丢状态、重复亮、重进后错乱。

### 4.4 在 UI 打开或预制体创建时顺手驱动红点数量

这也是错的。

红点数量应该由业务数据层在状态变化时计算出来，再统一调用红点赋值。  
UI 只负责拿当前红点状态做显示，不负责重新计算业务红点。

### 4.5 直接把所有细粒度节点先拍满

红点树要先保证层级正确，再补叶子。  
不要一上来几十个末级 key 全塞进去，最后发现父级设计错了又全部返工。
