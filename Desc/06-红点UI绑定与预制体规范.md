# 红点 UI 绑定与预制体规范

## 1. 文档目标

这份文档只回答一件事：

- UI 上怎么挂红点表现

它不负责解释：

- 红点 key 怎么分配
- 红点树怎么设计
- 红点数量谁来计算

这些内容分别看前面的文档。

## 2. 已有参考资源

参考目录：

- `Packages/cn.etetet.yiuireddot/Assets/GameRes/YIUI/RedDot/Bind`

该目录下当前有 3 个参考预制体，代表 3 种基础绑定方式。

### 2.1 `RedDot.prefab`

核心脚本：

- `RedDotBind`

特征：

- 基于 `MonoBehaviour`
- 绑定一个 red dot key 后，会自动监听红点变化
- 只负责显示/隐藏
- **不显示具体数字**

适用场景：

- 只需要“有红点就亮，没有红点就灭”的入口
- 不关心数量，只关心是否存在提醒

典型例子：

- 某个按钮角标点
- 某个页签角上的普通小红点

### 2.2 `RedDot_Text.prefab`

核心脚本：

- `RedDotTextBind`

特征：

- 继承自 `RedDotBind`
- 在显示/隐藏之外，增加数字显示
- 文本使用 UGUI 原生文本

适用场景：

- 需要显示红点数量
- 项目该处仍使用原生 UGUI 文本

典型例子：

- 邮件未读数
- 某类奖励累计可领取数

### 2.3 `RedDot_TMP.prefab`

核心脚本：

- `RedDotTmpBind`

特征：

- 功能上和 `RedDotTextBind` 类似
- 区别在于文本使用 TMP

适用场景：

- 需要显示数字
- 该界面文本体系是 TMP

## 3. 这 3 个预制体的定位

先把话说死：

- **这 3 个预制体是参考案例，不是建议直接拿来当正式业务资源使用**

原因有两个：

### 3.1 它们的定位就是包内参考

这几个预制体主要是告诉你：

- 脚本怎么挂
- 红点怎么绑定
- 有数字和无数字两种表现怎么做

它们更像“最小示例”，不是你项目业务 UI 的最终资产。

### 3.2 包升级可能覆盖

因为它们在包目录里：

- `Packages/cn.etetet.yiuireddot/...`

如果后续包升级、同步、替换，这些资源有被覆盖的风险。  
直接把业务正式资源建在这里，后面维护会很恶心。

## 4. 正确做法

正确方向是：

1. 参考这 3 个预制体的绑定方式
2. 在你自己的业务资源目录里创建对应的通用预制体
3. 业务界面真正使用你自己创建的预制体

也就是说：

- 包里的预制体是“样板”
- 你项目里的预制体才是“正式资源”

## 5. 推荐落地方式

### 5.1 先确定业务需要哪种表现

先判断：

- 只显示有/无
- 还是要显示数量
- 文本体系是 UGUI 还是 TMP

### 5.2 再复制设计思路，不复制依赖关系

你可以参考：

- 节点结构
- 脚本挂法
- 红点显示逻辑

但不要默认直接依赖包里的那份 prefab。

### 5.3 在项目业务目录创建自己的通用预制体

例如在你自己的 UI 资源目录下建：

- 某业务通用无数字红点 prefab
- 某业务通用数字红点 prefab
- 某业务 TMP 数字红点 prefab

后续哪里要用，就统一复用你自己的通用预制体。

## 6. 绑定方式说明

这 3 个脚本有共同点：

- 挂上去后
- 在 Inspector 里选择要监听的 red dot key
- 运行时会自动监听该 key 的变化
- 自己处理显示/隐藏，或数字刷新

因此它们的核心价值是：

- **自动监听**
- **自动解绑**
- **统一表现刷新**

## 7. 不只是静态绑定，也支持动态换绑

这里必须单独强调：

- 这些绑定脚本**不要求一定在 prefab 上预先把 key 选死**
- 它们支持运行时动态修改绑定的 red dot key

这意味着如果你的场景是动态需求，也可以做，不是只能靠预先在 Inspector 里选好固定 key。

### 7.1 适合动态换绑的场景

例如：

- 某个通用 UI 组件会被复用到不同入口
- 某个列表项或页签运行时才知道自己该监听哪个 key
- 同一个红点表现节点需要在不同状态下切换绑定目标

### 7.2 对应 API

核心 API 在：

- `Packages/cn.etetet.yiuireddot/Runtime/Bind/RedDotBind.cs`

方法：

```csharp
public void ChangeBind(int key, bool force = false)
```

作用：

- 运行时把当前绑定改成新的 red dot key

### 7.3 `ChangeBind` 的行为规则

#### 情况一：当前还没绑定 key

如果当前 `m_Key <= 0`，可以直接切到新 key。

#### 情况二：当前已经绑定了同一个 key

如果传入的 key 和当前一致，方法会直接返回，不重复处理。

#### 情况三：当前已经绑定了别的 key，且 `force = false`

这种情况下会报错并拒绝修改。

原因：

- 当前已经在监听一个红点
- 你又想切到另一个红点
- 但没有明确声明这是强制切换

#### 情况四：当前已经绑定了别的 key，且 `force = true`

会先解绑旧监听，再绑定到新的 key。

这是动态复用场景下的标准切换方式。

### 7.4 使用建议

如果是运行时动态复用同一个红点节点，推荐做法是：

1. 预制体初始 key 设为无
2. 在业务初始化或数据刷新时调用 `ChangeBind`
3. 如果节点会被重复复用到不同 key，切换时显式传 `force = true`

例如：

```csharp
redDotBind.ChangeBind(targetKey, true);
```

### 7.5 不要误用动态换绑

动态换绑解决的是“表现层监听哪个 key”的问题。  
它**不是**让你在 UI 层顺手创建业务红点逻辑的借口。

也就是说：

- 可以动态改绑定目标
- 但不能在 UI 层动态计算“这个 key 应该亮几个”

## 8. UI 层边界提醒

这里要再强调一次：

- 这些脚本是表现层绑定工具
- 它们不负责计算业务红点数量

不要把下面这种逻辑塞到 UI 绑定层：

- 打开界面时顺手去算当前主线任务红点有几个
- 刷一个 item 时顺手回推父级红点数量

UI 层只负责：

- 选中一个 key
- 监听这个 key
- 把结果显示出来

## 9. AI 后续处理 UI 红点时的默认规则

以后 AI 遇到“这里挂个红点”的请求时，默认按这套规则判断：

1. 先确认这里要不要数字
2. 再确认文本是 UGUI 还是 TMP
3. 优先在业务资源目录创建或复用正式通用 prefab
4. 不直接把包内参考 prefab 当正式资源落地
5. 判断这是静态绑定还是运行时动态换绑
6. 绑定只做显示，不做业务红点计算

## 10. 一句话总结

包里的 `RedDot / RedDot_Text / RedDot_TMP` 是**参考实现**。  
正式业务使用时，应在你自己的资源目录里创建对应通用预制体，再通过 `RedDotBind / RedDotTextBind / RedDotTmpBind` 绑定具体 key；如果需求是动态的，也可以在运行时调用 `ChangeBind(int key, bool force = false)` 切换绑定目标。
